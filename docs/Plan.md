### TC-01: Успешная оплата тура по одобренной карте

#### Тестовые данные

| Поле | Значение           | Примечание                                |
|------|--------------------|-------------------------------------------|
| Номер карты | 4444 4444 4444 4441 | Карта из эмулятора со статусом approved   |
| Месяц | Текущий месяц      | Валидный месяц - значение текущего месяца |
| Год | Текущий год        | Валидный год - значение текущего года     |
| Владелец | Ivan Petrov        | Валидное имя владельца на латинице        |
| CVC | 123                | Трёхзначное число (кроме значения 000)    |

#### Шаги

1. На странице покупки тура нажать на кнопку "Купить"
2. Заполнить поля формы тестовыми данными
3. Нажать на кнопку "Продолжить"
4. Выполнить запрос к БД для получения последнего платежа и проверки его статуса:  
`SELECT * FROM payment_entity ORDER BY created DESC LIMIT 1;`
5. Выполнить запрос к БД для получения последнего заказа и проверки его типа (обычный платёж):  
`SELECT * FROM order_entity ORDER BY created DESC LIMIT 1;`
6. По результатам шагов 4-5 сравнить значения колонки payment_id таблицы order_entity и transaction_id таблицы payment_entity
 
#### Ожидаемый результат

- В правом верхнем углу экрана появляется оповещение об успехе с текстом "Успешно" и "Операция одобрена Банком."
- Первый запрос к БД выполняется, запись существует и в колонке status значение APPROVED
- Второй запрос к БД выполняется, запись существует, в колонке credit_id значение NULL, в колонке 
payment_id присутствует значение
- Значения payment_id из таблицы order_entity и transaction_id из таблицы payment_entity совпадают

|-------------------|----------------------------|-------------------------------------------------------------------|
| **Java 11**       | Язык программирования      | Изучен в процессе обучения, широкие возможности для автоматизации |
| **IntelliJ IDEA** | Среда разработки           | Удобная IDE с поддержкой всех необходимых плагинов                |
| **Gradle**        | Система сборки             | Автоматизация сборки, управления зависимостями и запуска тестов   |
| **JUnit 5**       | Фреймворк для тестирования | Современный, гибкий, с поддержкой параметризации тестов           |
| **Selenide**      | Фреймворк для UI-тестов    | Простой синтаксис, автоматическое управление браузером            |
| **Docker**        | Контейнеризация окружения  | Изоляция сервисов, воспроизводимость окружения                    |
| **DBeaver**       | Работа с БД                | Удобный клиент для просмотра и проверки данных в БД               |
| **Allure**        | Формирование отчетности    | Наглядные отчеты с историей выполнения тестов                     |

## Оценка рисков при автоматизации

### 🔧 Технические риски

| Риск                                   | Описание                                                                                   | Вероятность | Влияние    |
|:---------------------------------------|:-------------------------------------------------------------------------------------------|:------------|:-----------|
| **Нестабильность тестового окружения** | Контейнеры могут падать, БД не успевать запускаться, что приводит к ложным падениям тестов | 🔴 Высокая  | 🟡 Среднее |
| **Проблемы с селекторами**             | Изменение верстки фронтенда сломает CSS-селекторы в тестах                                 | 🔴 Высокая  | 🔴 Высокое |
| **Конфликты портов**                   | Если порты 8080, 9999, 3306, 5432 заняты другими приложениями                              | 🟡 Средняя  | 🟡 Среднее |
| **Различия между СУБД**                | Запросы могут работать по-разному в MySQL и PostgreSQL                                     | 🟡 Средняя  | 🔴 Высокое |
| **Сложность отладки**                  | Ошибки в Docker окружении сложнее отлаживать, чем локальный запуск                         | 🟡 Средняя  | 🟡 Среднее |

### 📋 Процессные риски

| Риск                                     | Описание                                                                    | Вероятность | Влияние    |
|:-----------------------------------------|:----------------------------------------------------------------------------|:------------|:-----------|
| **Отсутствие требований и документации** | Что именно должно происходить при отказе банка? Какие сообщения показывать? | 🟡 Средняя  | 🔴 Высокое |
| **Отсутствие тестовых данных**           | Нужны разные карты для проверки граничных значений                          | 🟢 Низкая   | 🟡 Среднее |

### 💣 Риски самого приложения

| Риск                    | Описание                                                | Вероятность | Влияние     |
|:------------------------|:--------------------------------------------------------|:------------|:------------|
| **Безопасность данных** | Не должны сохраняться CVC-коды и полные данные карт     | ⚫ Критично  | ⚫ Критичное |
| **Обработка ошибок**    | Приложение должно корректно обрабатывать таймауты банка | 🟡 Средняя  | 🔴 Высокое  |
| **Целостность данных**  | Дублирование платежей при повторной отправке            | 🟡 Средняя  | ⚫ Критичное |
| **Производительность**  | При высокой нагрузке БД может не справляться            | 🟢 Низкая   | 🟡 Среднее  |

## Интервальная оценка с учётом рисков

### Этап 1: Подготовительный (8-10 часов)

| Задача                                | Время |
|---------------------------------------|:-----:|
| Настройка окружения в Docker          | 3-4 ч |
| Написание Dockerfile и docker-compose | 2-3 ч |
| Отладка ошибок запуска контейнеров    | 2-3 ч |
| Настройка подключения к БД из тестов  | 1-2 ч |

### Этап 2: Написание автотестов (24-48 часов)

| Категория    | Подкатегория                            | Время  |
|--------------|-----------------------------------------|:------:|
| **UI-тесты** | Позитивные сценарии                     | 6-8 ч  |
|              | Негативные сценарии                     | 8-10 ч |
|              | Проверка валидации полей                | 4-6 ч  |
| **БД-тесты** | Проверка сохранения данных              | 4-6 ч  |
|              | Проверка безопасности (CVC, маскировка) | 2-4 ч  |
|              | Кросс-СУБД тестирование                 | 4-6 ч  |
| **Отладка**  | Исправление ошибок в тестах             | 6-8 ч  |

### Итоговая оценка

| Уровень             | Часы  | Комментарий                       |
|:--------------------|:-----:|:----------------------------------|
| **Минимально**      | 32 ч  | Только критические сценарии       |
| **Оптимально**      | 48 ч  | Полное покрытие с запасом         |
| **С учётом рисков** | 58+ ч | При возникновении сложных проблем |
