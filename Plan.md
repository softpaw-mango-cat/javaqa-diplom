## Перечень автоматизируемых сценариев

### 🖥️ UI-тесты

### Тестирование формы оплаты по карте и в кредит


**Тестирование основных сценариев использования:**
- Покупка тура с одобренной валидной карты
- Покупка тура с отклоненной валидной карты
- Покупка тура с несуществующей карты (карты, которой нет в наборе карт)


**Тестирование валидации полей:**
- Поле номер карты: отправка формы с пустым значением поля, отправка формы с нулями, 
отправка формы с некорректным номером (меньше чем 16 цифр - 1 и 15)
- Поле месяц: отправка формы с пустым значением поля, отправка формы с нулями,
отправка формы с некорректными значениями - 1 (одно число), 13 (невадидный месяц)
- Поле год: отправка формы с пустым значением поля, отправка формы с нулями, 
отправка формы с одним значением - 1, отправка формы с невалидным годом (в прошлом) - 15, 
отправка формы с невалидным годом (в будущем, более 10 лет) - 40
- Поле владелец: отправка с пустым полем владелец, имя владельца на кириллице, 
слишком короткое имя (1 символ), слишком длинное имя (100 символов), спецсимволы в поле,
числа в поле
- Поле cvc: отправка с пустым полем cvc, нули в поле, 1 цифра в поле, 2 цифры в поле

### 🗄️ Тесты БД


**Проверка ответа от сервера при основных сценариях использования**


- Проверка ответа от сервера при покупке обычной одобренной картой
- Проверка ответа от сервера при покупке одобренной кредитной картой
- Проверка ответа от сервера при покупке обычной отклоненной картой
- Проверка ответа от сервера при покупке в кредит отклоненной картой

**Проверка сохранения корректных данных в БД MySQL и PostgreSQL**


- При обычной оплате одобренной картой сохраняются данные о покупке (таблица payment_entity)
- При обычной оплате одобренной картой сохраняется корректное значение статуса карты (статус approved)
- При обычной оплате одобренной картой сохраняется информация о транзации (проверка transaction_id)
- При обычной оплате одобренной картой сохраняется информация о заказе (таблица order_entity)
- При обычной оплате одобренной картой сохраняется корректная связь таблиц payment_entity и order_entity
- При обычной оплате одобренной картой нет записей в данных о кредите (таблица credit_request_entity)
- 

- При оплате в кредит одобренной картой сохраняются данные о покупке (таблица credit_request_entity)
- При оплате в кредит одобренной картой сохраняется корректное значение статуса карты (статус approved)
- При оплате в кредит одобренной картой сохраняется информация о транзации (проверка bank_id)
- При оплате в кредит одобренной картой сохраняется информация о заказе (таблица order_entity)
- При оплате в кредит одобренной картой сохраняется корректная связь таблиц credit_request_entity и order_entity
- При оплате в кредит одобренной картой нет записей в данных об обычной покупке (таблица payment_entity)
- 

- При попытке обычной оплаты отклоненной картой не сохраняются данные о покупке (таблица payment_entity)
- При попытке обычной оплаты отклоненной картой не сохраняются данные о заказе (таблица order_entity)
- При попытке обычной оплаты отклоненной картой не сохраняются данные о кредите (таблица credit_request_entity)


- При попытке оплаты в кредит отклоненной картой не сохраняются данные о покупке (таблица payment_entity)
- При попытке оплаты в кредит отклоненной картой не сохраняются данные о заказе (таблица order_entity)
- При попытке оплаты в кредит отклоненной картой не сохраняются данные о кредите (таблица credit_request_entity)

## Оценка рисков при автоматизации

### 🔧 Технические риски

| Риск                                   | Описание                                                                                   | Вероятность | Влияние    |
|:---------------------------------------|:-------------------------------------------------------------------------------------------|:------------|:-----------|
| **Нестабильность тестового окружения** | Контейнеры могут падать, БД не успевать запускаться, что приводит к ложным падениям тестов | 🔴 Высокая  | 🟡 Среднее |
| **Проблемы с селекторами**             | Изменение верстки фронтенда сломает CSS-селекторы в тестах                                 | 🔴 Высокая  | 🔴 Высокое |
| **Конфликты портов**                   | Если порты 8080, 9999, 3306, 5432 заняты другими приложениями                              | 🟡 Средняя  | 🟡 Среднее |
| **Различия между СУБД**                | Запросы могут работать по-разному в MySQL и PostgreSQL                                     | 🟡 Средняя  | 🔴 Высокое |
| **Сложность отладки**                  | Ошибки в Docker окружении сложнее отлаживать, чем локальный запуск                         | 🟡 Средняя  | 🟡 Среднее |

### 📋 Процессные риски

| Риск                                     | Описание                                                                    | Вероятность | Влияние    |
|:-----------------------------------------|:----------------------------------------------------------------------------|:------------|:-----------|
| **Отсутствие требований и документации** | Что именно должно происходить при отказе банка? Какие сообщения показывать? | 🟡 Средняя  | 🔴 Высокое |
| **Отсутствие тестовых данных**           | Нужны разные карты для проверки граничных значений                          | 🟢 Низкая   | 🟡 Среднее |

### 💣 Риски самого приложения

| Риск                    | Описание                                                | Вероятность | Влияние     |
|:------------------------|:--------------------------------------------------------|:------------|:------------|
| **Безопасность данных** | Не должны сохраняться CVC-коды и полные данные карт     | ⚫ Критично  | ⚫ Критичное |
| **Обработка ошибок**    | Приложение должно корректно обрабатывать таймауты банка | 🟡 Средняя  | 🔴 Высокое  |
| **Целостность данных**  | Дублирование платежей при повторной отправке            | 🟡 Средняя  | ⚫ Критичное |
| **Производительность**  | При высокой нагрузке БД может не справляться            | 🟢 Низкая   | 🟡 Среднее  |

## Интервальная оценка с учётом рисков

### Этап 1: Подготовительный (8-10 часов)

| Задача                                | Время |
|---------------------------------------|:-----:|
| Настройка окружения в Docker          | 3-4 ч |
| Написание Dockerfile и docker-compose | 2-3 ч |
| Отладка ошибок запуска контейнеров    | 2-3 ч |
| Настройка подключения к БД из тестов  | 1-2 ч |

### Этап 2: Написание автотестов (24-48 часов)

| Категория    | Подкатегория                            | Время  |
|--------------|-----------------------------------------|:------:|
| **UI-тесты** | Позитивные сценарии                     | 6-8 ч  |
|              | Негативные сценарии                     | 8-10 ч |
|              | Проверка валидации полей                | 4-6 ч  |
| **БД-тесты** | Проверка сохранения данных              | 4-6 ч  |
|              | Проверка безопасности (CVC, маскировка) | 2-4 ч  |
|              | Кросс-СУБД тестирование                 | 4-6 ч  |
| **Отладка**  | Исправление ошибок в тестах             | 6-8 ч  |

### Итоговая оценка

| Уровень             | Часы  | Комментарий                       |
|:--------------------|:-----:|:----------------------------------|
| **Минимально**      | 32 ч  | Только критические сценарии       |
| **Оптимально**      | 48 ч  | Полное покрытие с запасом         |
| **С учётом рисков** | 58+ ч | При возникновении сложных проблем |
